ПОДРОБНЫЙ РАЗБОР АРХИТЕКТУРЫ ПРОЕКТА "ЕЖОВЫЕ РУКИ"

====================
1. ОБЩАЯ ИДЕЯ СИСТЕМЫ
====================

Проект "Ежовые руки" - это веб-приложение для взаимодействия родителей и помощников.

В системе есть три роли:
- Родитель - добавляет карточки детей, ищет помощников по объявлениям, отправляет заявки и отслеживает их статусы.
- Помощник - создает объявления о своих услугах, получает отклики родителей, берет заявки в работу, завершает или отклоняет их.
- Администратор - управляет пользователями и категориями.

Основной сценарий:
1. Пользователь регистрируется как родитель или помощник.
2. Помощник создает объявление с категорией, описанием и ценой.
3. Родитель добавляет ребенка и ищет подходящее объявление.
4. Родитель отправляет заявку на объявление.
5. Помощник видит заявку в очереди и меняет ее статус:
   - Новая
   - В работе
   - Выполнена
   - Отклонена
6. Администратор следит за пользователями и справочником категорий.

Проект разделен на две независимые части:
- client/ - фронтенд на React.
- server/ - бэкенд на NestJS.


====================
2. СТЕК И ИНСТРУМЕНТЫ
====================

ФРОНТЕНД:
- React 19 - библиотека для построения пользовательского интерфейса.
- TypeScript - строгая типизация кода.
- Vite 8 - быстрый dev-сервер и сборщик фронтенда.
- React Router DOM 7 - маршрутизация между страницами.
- TanStack React Query 5 - загрузка, кэширование и обновление серверных данных.
- Ant Design 6 - UI-библиотека: формы, таблицы, кнопки, модальные окна, селекты, пагинация, теги.
- ESLint - проверка качества кода.

Что особенно важно на фронтенде:
- Все запросы к серверу вынесены в client/src/api.
- Авторизация хранится в AuthContext.
- Роли проверяются через RequireAuth и RequireRole.
- Большая часть интерфейса строится на Ant Design, а внешний вид дополнительно настраивается через ConfigProvider и CSS.

БЭКЕНД:
- NestJS 11 - серверный фреймворк поверх Node.js.
- TypeScript - основной язык бэкенда.
- TypeORM - ORM для работы с базой данных через сущности и репозитории.
- PostgreSQL - база данных.
- @nestjs/config - работа с переменными окружения из .env.
- @nestjs/jwt - выпуск JWT-токенов.
- Passport + passport-jwt - проверка JWT при защищенных запросах.
- bcrypt - хэширование паролей.
- class-validator + class-transformer - валидация входящих DTO.
- @nestjs/swagger + swagger-ui-express - Swagger-документация API.
- Jest - тестовый фреймворк.
- ESLint + Prettier - проверка и форматирование кода.

Что особенно важно на бэкенде:
- Сервер принимает запросы с префиксом /api.
- Глобальный ValidationPipe отбрасывает лишние поля и возвращает ProblemDetails-ошибки.
- JWT используется для авторизации пользователя и определения его роли.
- Ролевая защита строится на RolesGuard и декораторе @Roles.
- База данных описана через TypeORM entity-классы.
- Миграции лежат в server/src/migrations и применяются автоматически при старте.

3. КОРЕНЬ ПРОЕКТА
====================

Основные элементы корня:
- client/ - фронтенд-приложение.
- server/ - серверное приложение.
- PROJECT_ARCHITECTURE.txt - этот файл с разбором проекта.
- .git/ - история изменений проекта.

На практике разработка обычно идет так:
1. Запускается сервер:
   cd server
   npm.cmd run start:dev

2. Запускается клиент:
   cd client
   npm.cmd run dev

3. Клиент открывается в браузере на Vite-порту, обычно:
   http://localhost:5173
   или
   http://127.0.0.1:5173

4. Клиент отправляет HTTP-запросы на сервер:
   http://localhost:3000/api/...

====================
11. СВЯЗЬ ФРОНТЕНДА И БЭКЕНДА
====================

Типичный путь данных:
1. Пользователь нажимает кнопку на странице React.
2. Страница вызывает функцию из client/src/api.
3. api-функция использует apiRequest из http.ts.
4. Запрос уходит на server: http://localhost:3000/api/...
5. Nest Controller принимает HTTP-запрос.
6. ValidationPipe проверяет DTO.
7. Controller вызывает Service.
8. Service работает с TypeORM Repository и базой данных.
9. Mapper превращает Entity в DTO ответа.
10. Клиент получает JSON.
11. React Query обновляет кэш.
12. Компонент перерисовывается.

Пример регистрации:
- client/src/pages/public/RegisterPage.tsx
- client/src/context/AuthContext.tsx
- client/src/api/auth.ts -> registerRequest
- server/src/auth/auth.controller.ts -> register
- server/src/auth/dto/register.dto.ts
- server/src/auth/auth.service.ts
- server/src/users/users.service.ts
- server/src/users/user.entity.ts

Пример создания заявки:
- client/src/pages/parent/SearchPage.tsx
- client/src/api/requests.ts -> createRequest
- server/src/requests/requests.controller.ts
- server/src/requests/dto/create-request.dto.ts
- server/src/requests/requests.service.ts
- server/src/requests/request.entity.ts
- server/src/requests/requests.mapper.ts

13. КЛЮЧЕВЫЕ АРХИТЕКТУРНЫЕ ПРАВИЛА ПРОЕКТА
====================

1. Страницы фронтенда не должны напрямую писать fetch.
   Для запросов есть client/src/api.

2. Бизнес-логика должна жить на сервере в service-классах.
   Controller должен принимать запрос, валидировать DTO и вызывать сервис.

3. DTO отделяют внешний API от внутренних Entity.
   Клиент не обязан знать все поля таблиц базы данных.

4. Mapper нужен для преобразования Entity -> ResponseDto.
   Это помогает не отдавать лишние данные, например passwordHash.

5. Проверка ролей должна быть на сервере.
   Фронтенд может скрыть меню, но настоящая защита находится в Guards.

6. Все ошибки сервера желательно возвращать в едином формате ProblemDetails.
   Клиент уже умеет читать title/detail.

7. Пагинация должна быть серверной там, где список потенциально большой.
   На фронтенде отображается только текущая страница данных.

8. UI строится на Ant Design.
   Кастомный CSS используется для темы, компоновки и аккуратной стилизации, но базовые элементы остаются компонентами библиотеки.
